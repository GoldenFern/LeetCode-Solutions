# LeetCode 1: 两数之和 (Two Sum)

## 1. 题目描述

给定一个整数数组 `nums` 和一个整数目标值 `target`，请你在该数组中找出 **和为目标值** `target` 的那 **两个** 整数，并返回它们的数组下标。

你可以假设每种输入只会对应一个答案。但是，数组中同一个元素在答案里不能重复出现。

**示例 1:**

> 输入：nums = [2,7,11,15], target = 9
> 输出：[0,1]

**提示：**

* $2 \le nums.length \le 10^4$
* $-10^9 \le nums[i] \le 10^9$
* 只会存在一个有效答案

---

## 2. 算法基本原理：哈希表 (Hash Table)

本题的核心在于将“搜索问题”转化为“查表问题”。

* **暴力解法**：双重循环遍历所有组合，时间复杂度 $O(n^2)$。
* **哈希优化**：遍历数组时，将每个数及其索引存入哈希表。对于当前数 `x`，直接在表中查询 `target - x` 是否存在。
* **复杂度**：时间 $O(n)$，空间 $O(n)$。

---

## 3. 深度解析：哈希表的底层原理与 Python 实现

为什么哈希表能实现 $O(1)$ 的查找？这背后涉及三个核心机制：

### A. 哈希函数 (Hash Function) 与 随机访问

哈希表本质上是一个**数组**。

1. **计算哈希值**：通过 `hash(key)` 将任意可哈希对象转为一个巨大的整数。
2. **映射索引**：将该整数对数组长度取模（或进行位运算 `hash & (mask)`），得到数组下标。
3. **$O(1)$ 来源**：由于数组在内存中是连续的，计算机通过 `地址 = 首地址 + 索引 * 偏移量` 可以在常数时间内定位到任何位置。

### B. 哈希冲突 (Collision) 的处理

当两个不同的 Key 映射到同一个索引时，就会发生冲突。主流有两种解决方法：

1. **拉链法 (Chaining)**：每个桶（Bucket）维护一个链表，冲突的元素挂在链表后（Java `HashMap` 采用此法）。
2. **开放寻址法 (Open Addressing)**：如果位置被占用，按照某种探测序列（Probing）寻找下一个空位。**Python 的 `dict` 采用此法**，它使用一种伪随机探测序列来减少聚集现象。

### C. Python `dict` 的内存优化 (Python 3.6+)

传统的哈希表为了减少冲突，通常会预留大量空位（负载因子通常控制在 2/3 以下），这非常浪费内存。

* **现代 Python 实现**：将哈希表拆分为两个数组：
  * `Indices`：一个紧凑的稀疏数组，存储 `Entries` 的下标。
  * `Entries`：一个紧凑的稠密数组，按插入顺序存储 `[hash, key, value]`。
* **优点**：这种设计不仅节省了内存，还让 Python 的字典变得**有序**（按插入顺序遍历）。

### D. 扩容 (Resizing)

当哈希表变得拥挤（负载因子过高）时，查找效率会下降。此时 Python 会申请一个更大的内存空间（通常是原容量的 2-4 倍），并将所有旧元素重新哈希（Rehash）到新表中。虽然单次扩容是 $O(n)$，但分摊到每次插入操作上，依然是**均摊 $O(1)$**。

## 4. 编程中需要注意的小细节 (易错点)

1. **一次遍历 (One-pass) 的逻辑顺序**：
   
   * **错误做法**：先将所有元素存入哈希表，再遍历查找。这会导致处理 `target = 6, nums = [3, 2, 4]` 时，3 会匹配到它自己。
   * **正确做法**：边查找边入库。在处理当前元素 `x` 时，只在它**之前**的元素中找补数。这样既能避免重复利用同一个元素，也处理了数组中有重复数字的情况。

2. **哈希表的 Key 必须可哈希**：
   
   * 在 Python 中，`list` 是不可哈希的。如果题目要求查找的是一组数字的组合（如 `[1, 2]`）是否存在，必须先将其转为 `tuple`。

3. **内存开销**：
   
   * 虽然哈希表时间快，但在处理海量数据（如数亿个整数）时，哈希表的内存占用远高于原始数组。在内存受限的嵌入式场景中，可能需要权衡是否使用排序+双指针（$O(n \log n)$ 时间，$O(1)$ 空间）。
